## Overview
Two parallel delivery mechanisms that must stay equivalent:
- `kustomize/` — kubebuilder-style tree (`make install` / `make deploy`)
- `doplane/` — Helm chart (`make helm-lint` / `helm-template` / `helm-package`)

## Directory structure
```
deploy/
  kustomize/
    crd/bases/         → generated by `make manifests` (do not hand-edit)
    rbac/role.yaml     → generated from +kubebuilder:rbac markers (do not hand-edit)
    manager/           → Deployment; runner_image_patch.yaml is REWRITTEN by
                         `make set-runner-image` from RUNNER_IMG
                         (dependency of deploy/build-installer)
    default/           → top-level kustomization
    samples/           → CR samples (pin package versions to Dockerfile.runner ARGs)
  doplane/
    crds/              → SYNCED from kustomize CRDs by `make sync-helm-crds`
                         (do not hand-edit)
    templates/         → Helm templates
    values.yaml        → chart defaults (manager image, runner image, …)
```

## Rules
- Generated files (`crd/bases/`, `rbac/role.yaml`, Helm `crds/`) are outputs
  of `make manifests` — change the Go types/markers and regenerate instead.
- Manager image is set via `kustomize edit set image controller=$(IMG)`; the
  runner image via the `RUNNER_IMAGE` env, rewritten from `$(RUNNER_IMG)` by
  `make set-runner-image`. Never hard-code image tags in `manager.yaml`.
- Keep kustomize and the Helm chart equivalent: if you change one, mirror the
  other (or extend the Makefile sync targets).
- Runner pods are hardened by the JobRunner in Go
  (`internal/pulumido/job.go`), not by these manifests — pod-security changes
  for runner Jobs belong there.
- Local test flow: build both images (`make docker-build docker-build-runner`),
  `kind load docker-image …`, `make install deploy IMG=… RUNNER_IMG=…`; AWS
  credentials for runner pods come from `hack/sync-creds.sh` (Pulumi ESC,
  short-lived — re-run when expired).

## Setup & environment
- Tools arrive via the Makefile (`bin/kustomize`, controller-gen); Helm from PATH
- Cluster for local testing: kind (`kind create cluster --name doplane`)

## Build & tests
- Render checks: `make helm-lint`, `make helm-template`, `bin/kustomize build deploy/kustomize/default`
- Full delivery test: build + `kind load` images, `make install deploy IMG=… RUNNER_IMG=…`, apply `examples/`

## Code style & conventions
- Kustomize patches over forked copies; strategic-merge patches live next to the base they modify
- Chart values mirror kustomize knobs (image, runner image) — keep names consistent

## Security & safety
- No credentials in manifests; runner creds only via the optional `provider-credentials` Secret
- Manager stays distroless, non-root, read-only rootfs; don't relax pod security to "fix" issues

## PR/commit checklist
- [ ] `make manifests` run after any marker/type change; no manual edits to generated files
- [ ] `make helm-lint` and `bin/kustomize build deploy/kustomize/default` both succeed
- [ ] Sample CRs still pin provider versions matching `Dockerfile.runner` ARGs

## Patterns to Follow
> Mirror existing kustomize/Helm structures; check `deploy/kustomize/manager/runner_image_patch.yaml` for the image-plumbing pattern.
